# Performance and Correctness Report

Test date: 2026-07-17

Environment: local macOS development machine, Spring Boot 2.3.12 on JDK 11,
MySQL and Redis in Docker. HTTP load was generated by
`benchmarks/http_benchmark.py`. Every comparison used the same machine,
dataset, concurrency, and test client.

## Cache benchmark

Workload: 10,000 reads of shop `1`, concurrency 100, after 50 warm-up reads.

| Path | QPS | P95 | P99 |
| --- | ---: | ---: | ---: |
| Direct MySQL | 5,362.81 | 32.89 ms | 43.88 ms |
| Redis only | 6,313.59 | 28.73 ms | 36.56 ms |
| Caffeine + Redis | 8,232.77 | 22.51 ms | 30.31 ms |

- Multi-level cache improved QPS by 53.52% over direct MySQL and reduced P99
  by 30.93%.
- Caffeine improved QPS by 30.40% over Redis-only and reduced P99 by 17.10%.
- The warmed single-hot-key phase recorded 10,000 Caffeine hits, zero Redis
  reads in the production query path, zero database loads, and a 100% measured
  cache hit rate. “100%” applies only to this controlled warmed workload.
- Raw result: `results/cache.json`.

## Seckill benchmark

Workload: 1,000 distinct authenticated users competing for 200 units,
concurrency 100.

| Path | QPS | P95 | P99 | Accepted |
| --- | ---: | ---: | ---: | ---: |
| Synchronous MySQL | 861.45 | 312.50 ms | 408.81 ms | 200 |
| Redis Lua + Stream | 3,668.06 | 58.36 ms | 68.59 ms | 200 |

- The asynchronous endpoint delivered 4.26x throughput and reduced P99 by
  83.22% compared with the synchronous baseline.
- The Stream consumer persisted the 200 accepted orders at 309.15 orders/s.
- Validation after consumption: 200 orders, 200 distinct users, zero
  oversold orders, zero duplicate orders, MySQL stock 0, Redis stock 0,
  and Pending List size 0.
- Raw results: `results/seckill.json` and `results/seckill-sync.json`.

## Order-state correctness

- 100 simultaneous payment-versus-close rounds produced zero illegal states
  and zero rounds where both or neither CAS update won.
- Running the close operation twice restored MySQL and Redis stock exactly
  once.
- The scheduled task runs every 60 seconds and processes at most 100 expired
  orders per batch; therefore its configured closure bound for a batch is
  60 seconds.
- Raw results: `results/order-cas.json` and `results/auto-close.json`.

## Reproduce

```bash
python3 benchmarks/http_benchmark.py cache \
  --requests 10000 --concurrency 100 --shop-id 1

python3 benchmarks/http_benchmark.py seckill \
  --requests 1000 --concurrency 100 --voucher-id 90001 \
  --tokens tokens.txt
```

The benchmark voucher and tokens are test fixtures. Create/reset the voucher
before a rerun, and verify order counts, distinct users, both stock values, and
Stream Pending count after the run.
