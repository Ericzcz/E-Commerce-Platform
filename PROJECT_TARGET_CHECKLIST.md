## 2026-07-17 最终实施结果

本轮已完成原清单中的核心功能：

- ✅ Redis Cache Aside、空值缓存、统一逻辑过期格式、Redisson 锁和有界线程池异步重建。
- ✅ Caffeine → Redis → MySQL 两级查询，Redis Lua 滑动窗口热点识别，本地缓存跨实例失效通知。
- ✅ 数据更新成功后删除缓存；失败失效进入重试队列，TTL 作为最终兜底。
- ✅ Redis Lua 原子库存预扣与一人一单、Redis Stream 异步订单、MySQL 唯一索引和条件扣库存。
- ✅ Consumer Group、事务后 ACK、Pending 恢复、最多 5 次重试、死信 Stream 和库存/资格补偿。
- ✅ SpringTask 超时关单、分页限量扫描、MySQL/Redis 库存补偿以及支付/关单状态 CAS。
- ✅ 自动化并发正确性测试和可重复的 HTTP 基准脚本。

实测数据（本机、JDK 11、MySQL/Redis Docker）：

- 缓存：10,000 请求、并发 100。MySQL 直查 5,362.81 QPS、P99 43.88 ms；
  Redis 单级 6,313.59 QPS、P99 36.56 ms；Caffeine + Redis 8,232.77 QPS、
  P99 30.31 ms。
- 热点预热阶段整体命中率 100%，10,000 次请求全部命中 Caffeine，Redis
  请求和数据库回源均为 0。该比例只代表单热点、充分预热的受控测试。
- 秒杀：1,000 个不同用户并发 100 抢 200 份库存，异步接口 3,668.06 QPS、
  P99 68.59 ms；最终 200 单、重复 0、超卖 0、Redis/MySQL 库存均为 0。
- 同步基线为 861.45 QPS、P99 408.81 ms；异步版本吞吐量为 4.26 倍，
  P99 降低 83.22%，消费者落库吞吐量 309.15 单/秒。
- 100 轮支付与关单并发竞争中非法状态 0、双赢家/双输家 0；重复执行关单
  只补偿一次库存。定时任务每 60 秒执行、每批最多处理 100 单。

## 可直接用于简历的版本

**缓存优化：** 基于 Redis 实现 Cache Aside、空值缓存与逻辑过期，使用
Redisson 分布式锁和有界线程池异步重建热点缓存；在并发 100、10,000 次
热点查询中，相比 MySQL 直查将 P99 从 43.88 ms 降至 30.31 ms，吞吐量从
5,362.81 QPS 提升至 8,232.77 QPS。

**多级缓存：** 使用 Caffeine 与 Redis 构建两级缓存，并通过 Redis Lua
滑动窗口动态识别热点；在充分预热的单热点测试中实现 100% 缓存命中，
相较 Redis 单级缓存将吞吐量从 6,313.59 QPS 提升至 8,232.77 QPS，
P99 从 36.56 ms 降至 30.31 ms。

**秒杀与异步订单：** 使用 Redis Lua 原子完成库存预扣和一人一单校验，
通过 Redis Stream Consumer Group 异步落库并实现 ACK、Pending 重试、
死信和幂等消费；1,000 个用户并发抢购 200 份库存时达到 3,668.06 QPS，
超卖和重复订单均为 0，P99 相比同步落库从 408.81 ms 降至 68.59 ms。

**订单状态管理：** 使用 SpringTask 分批关闭超时未支付订单并补偿库存，
通过条件更新 CAS 解决支付与关单竞争；100 轮并发测试中非法订单状态为
0，重复关单库存补偿为 0。
